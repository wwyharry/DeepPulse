## 1. 项目概述

### 1.1 项目定位

本项目是一个面向 A 股市场的日 K 级别数据采集、存储与智能分析系统，核心设计目标：

- **多源冗余**: 通过多个独立数据源交叉验证，最大限度保证数据完整性
- **本地存储**: DuckDB 嵌入式数据库，零运维成本，单文件部署
- **Agent 友好**: 提供结构化查询接口，可直接被 LLM Agent 调用
- **断点续传**: 采集过程支持中断恢复，适合大规模长时间任务
- **AI 分析**: 内置短线分析 AI Agent，基于 DeepSeek 等大模型自动分析股票走势
- **流式输出**: 支持流式输出与 thinking 推理过程展示，实时反馈分析进展

### 1.2 数据范围

| 维度 | 范围 |
|------|------|
| 市场 | 沪深主板（沪市 6 开头、深市 0 开头，排除科创板 688、创业板 3、北交所 8） |
| 时间 | 过去 5 年（可通过配置修改） |
| 级别 | 日 K 级别（OHLCV + 成交额 + 换手率） |
| 预计规模 | ~3500 只股票 × 5 年 × ~250 交易日/年 ≈ 430 万行 |

## 4. 数据库设计

### 4.1 数据库选型：DuckDB

选择理由：
- **列式存储**: 分析查询性能远超 SQLite/MySQL
- **嵌入式**: 零配置，单文件，无需安装服务
- **SQL 兼容**: 支持标准 SQL，学习成本低
- **Python 原生**: 与 pandas 深度集成，可直接查询 DataFrame
- **并发安全**: 支持单写多读

### 4.2 表结构

#### stock_info（股票基本信息）

```sql
CREATE TABLE stock_info (
    code        VARCHAR PRIMARY KEY,  -- 股票代码（6位，如 600000）
    name        VARCHAR,              -- 股票名称
    market      VARCHAR,              -- 市场标识：sh（上海）/ sz（深圳）
    board       VARCHAR,              -- 板块：main（主板）
    list_date   DATE,                 -- 上市日期
    delist_date DATE,                 -- 退市日期（可为 NULL）
    updated_at  TIMESTAMP DEFAULT current_timestamp  -- 记录更新时间
);
```

**设计说明**:
- `code` 使用 6 位纯数字字符串，不包含市场前缀
- `market` 字段区分上海/深圳，便于生成不同数据源的代码格式
- 多源采集时，通过 DELETE + INSERT 实现更新（保留最新数据）

#### daily_kline（日 K 线数据 - 主表）

```sql
CREATE TABLE daily_kline (
    code        VARCHAR NOT NULL,     -- 股票代码
    trade_date  DATE    NOT NULL,     -- 交易日期
    open        DOUBLE,              -- 开盘价（前复权）
    high        DOUBLE,              -- 最高价
    low         DOUBLE,              -- 最低价
    close       DOUBLE,              -- 收盘价
    volume      BIGINT,              -- 成交量（股）
    amount      DOUBLE,              -- 成交额（元）
    turnover    DOUBLE,              -- 换手率（%，如 1.5 表示 1.5%）
    data_source VARCHAR NOT NULL,    -- 数据来源标识（akshare / baostock）
    PRIMARY KEY (code, trade_date, data_source)
);
```

**设计说明**:
- 主键为 `(code, trade_date, data_source)` 三元组，同一股票同日可存多个源的数据
- 价格为前复权价格，便于技术分析计算
- `data_source` 字段支持多源数据共存，可按源过滤或做一致性校验
- `INSERT OR IGNORE` 保证重复采集不会产生重复数据

#### fetch_log（采集日志）

```sql
CREATE TABLE fetch_log (
    id          INTEGER PRIMARY KEY,  -- 自增 ID
    code        VARCHAR,             -- 股票代码
    data_source VARCHAR,             -- 数据源名称
    start_date  DATE,                -- 本次采集起始日期
    end_date    DATE,                -- 本次采集截止日期
    row_count   INTEGER,             -- 本次写入行数
    status      VARCHAR,             -- success / failed
    error_msg   VARCHAR,             -- 失败原因（可为 NULL）
    fetched_at  TIMESTAMP DEFAULT current_timestamp  -- 采集时间
);
```

**设计说明**:
- 用于断点续传：查询某股票某源的最后成功采集日期，下次从该日期 +1 天继续
- 用于数据质量审计：可统计失败率、各源写入量等

#### long_term_memories（长期记忆）

```sql
CREATE TABLE long_term_memories (
    id                  VARCHAR PRIMARY KEY,   -- UUID
    memory_type         VARCHAR NOT NULL,      -- preference/insight/fact/context/summary/learning
    content             VARCHAR NOT NULL,      -- 记忆内容
    keywords_json       VARCHAR NOT NULL,      -- JSON 数组，用于检索
    tags_json           VARCHAR NOT NULL,      -- JSON 数组，分类标签
    importance          FLOAT DEFAULT 0.5,     -- 重要度 0.0-1.0
    access_count        INTEGER DEFAULT 0,     -- 访问次数
    last_accessed_at    TIMESTAMP,             -- 最后访问时间（衰减起点）
    decay_halflife_days INTEGER DEFAULT 30,    -- 衰减半衰期（天）
    source_session_id   VARCHAR,               -- 来源会话 ID
    source_tool         VARCHAR,               -- 来源工具名
    compressed_from     VARCHAR,               -- 压缩来源
    is_archived         BOOLEAN DEFAULT FALSE,  -- 是否归档（软删除）
    learned_what        VARCHAR DEFAULT '',     -- 学到了什么（learning 类型）
    learned_why         VARCHAR DEFAULT '',     -- 为什么重要（learning 类型）
    apply_when          VARCHAR DEFAULT '',     -- 何时应用（learning 类型）
    created_at          TIMESTAMP DEFAULT current_timestamp,
    updated_at          TIMESTAMP DEFAULT current_timestamp
);
```

**设计说明**:
- 6 种记忆类型，各有不同半衰期：preference(90天), insight(30天), fact(60天), context(14天), summary(45天), learning(180天)
- `learning` 类型是最宝贵的记忆，来自用户教学和纠错，半衰期最长
- 衰减公式：`effective_weight = importance * exp(-0.693 * days / half_life)`
- `learned_what/learned_why/apply_when` 为 learning 类型的结构化字段
- 访问会重置衰减时钟（更新 `last_accessed_at`）
- `embedding` 列存储 768 维 float32 向量（BLOB），用于语义搜索

**新增列（v4.0 迁移）**:
```sql
ALTER TABLE long_term_memories ADD COLUMN IF NOT EXISTS embedding BLOB;
```

#### predictions（投资预测记录）

```sql
CREATE TABLE IF NOT EXISTS predictions (
    id                VARCHAR PRIMARY KEY,    -- UUID
    stock_code        VARCHAR NOT NULL,       -- 股票代码
    stock_name        VARCHAR,                -- 股票名称
    prediction_type   VARCHAR,                -- 'direction'/'price_range'/'pattern'
    direction         VARCHAR,                -- 'bullish'/'bearish'/'neutral'
    target_price      DOUBLE,                 -- 目标价
    stop_loss         DOUBLE,                 -- 止损价
    timeframe_days    INTEGER,                -- 预测时间窗口（天）
    reasoning         TEXT,                   -- 预测理由
    confidence        FLOAT,                  -- 置信度 0.0-1.0
    memory_ids_json   VARCHAR,                -- 关联记忆 ID 列表（JSON）
    created_at        TIMESTAMP,              -- 创建时间
    check_after_date  DATE,                   -- 验证日期
    actual_direction  VARCHAR,                -- 实际方向
    actual_price      DOUBLE,                 -- 实际价格
    actual_return_pct FLOAT,                  -- 实际收益率
    outcome           VARCHAR DEFAULT 'pending', -- 'correct'/'wrong'/'partial'/'pending'
    outcome_notes     TEXT,                   -- 验证备注
    checked_at        TIMESTAMP,              -- 验证时间
    learning_extracted BOOLEAN DEFAULT FALSE  -- 是否已提取教训
);
```

**设计说明**:
- Agent 每次给出分析结论时自动保存预测
- 下次分析同只股票时自动调用 `check_predictions` 验证
- 正确预测 → 关联记忆 importance +0.1；错误预测 → importance -0.1
- `outcome` 四种状态：pending(待验证)、correct(正确)、wrong(错误)、partial(部分正确)

#### knowledge_entities（知识图谱实体）

```sql
CREATE TABLE IF NOT EXISTS knowledge_entities (
    id              VARCHAR PRIMARY KEY,    -- UUID
    entity_type     VARCHAR NOT NULL,       -- 'stock'/'indicator'/'pattern'/'strategy'/'concept'/'rule'
    name            VARCHAR NOT NULL,       -- 实体名称（如 "RSI超卖"、"低吸战法"）
    attributes_json TEXT,                   -- 属性 JSON（如 {"threshold": 20}）
    created_at      TIMESTAMP,
    updated_at      TIMESTAMP
);
```

#### knowledge_relations（知识图谱关系）

```sql
CREATE TABLE IF NOT EXISTS knowledge_relations (
    id                  VARCHAR PRIMARY KEY,    -- UUID
    source_id           VARCHAR NOT NULL,       -- 源实体 ID
    target_id           VARCHAR NOT NULL,       -- 目标实体 ID
    relation_type       VARCHAR NOT NULL,       -- 'triggers'/'requires'/'contradicts'/'supports'/'applies_to'/'user_prefers'
    weight              FLOAT DEFAULT 1.0,      -- 关系权重
    evidence_memory_ids VARCHAR,                -- 支撑此关系的记忆 ID（JSON）
    created_at          TIMESTAMP
);
```

**设计说明**:
- 实体类型：stock(股票)、indicator(指标)、pattern(形态)、strategy(策略)、concept(概念)、rule(规则)
- 关系类型：triggers(触发)、requires(需要)、contradicts(矛盾)、supports(支持)、applies_to(适用于)、user_prefers(用户偏好)
- 示例：RSI超卖 --triggers--> 低吸战法 (weight=0.8)
- 保存记忆时自动提取实体和关系，分析时查询关联实体辅助推理

#### user_profile（用户交易画像）

```sql
CREATE TABLE IF NOT EXISTS user_profile (
    key               VARCHAR PRIMARY KEY,    -- 画像键（如 'trading_style'）
    value             TEXT,                   -- 画像值（如 '低吸型'）
    confidence        FLOAT,                  -- 置信度（基于对话次数推断）
    source_memory_ids VARCHAR,                -- 来源记忆 ID（JSON）
    updated_at        TIMESTAMP
);
```

**设计说明**:
- 自动从对话中提取用户交易特征
- 画像键包括：trading_style(交易风格)、risk_tolerance(风险偏好)、stop_loss_habit(止损习惯)、preferred_indicators(偏好指标)、watched_sectors(关注板块)、watched_stocks(关注股票)
- 置信度基于提取次数累积，多次确认的画像更可靠

### 4.3 索引策略

DuckDB 自动为主键创建索引。当前主键已覆盖常用查询模式：
- 按股票查日 K: `(code, trade_date)` 命中主键前缀
- 按日期范围查: `(trade_date)` 可利用主键

如需优化特定查询，可手动添加索引：
```sql
CREATE INDEX idx_kline_date ON daily_kline(trade_date);
CREATE INDEX idx_kline_source ON daily_kline(data_source);
```

## 7. 采集机制详解

### 7.1 采集流程

```
开始
  │
  ├─ 1. 初始化数据库（init_tables）
  │
  ├─ 2. 采集股票列表
  │     ├─ 遍历所有数据源
  │     ├─ 各源独立获取 → 合并去重
  │     └─ 写入 stock_info 表
  │
  ├─ 3. 采集日 K 数据
  │     ├─ 遍历每只股票
  │     ├─ 检查断点续传位置（fetch_log）
  │     ├─ 调用数据源 API
  │     ├─ 写入 daily_kline 表
  │     ├─ 记录采集日志
  │     └─ 等待限流间隔
  │
  └─ 4. 输出统计报告
```

### 7.2 断点续传机制

```python
# 核心逻辑（collector.py:fetch_kline_for_stock）
last_date = get_last_fetch_date(conn, code, source.name)
if last_date:
    last = date.fromisoformat(last_date)
    if last >= end:
        return 0  # 已完成，跳过
    actual_start = last + timedelta(days=1)  # 从次日继续
```

- 判断依据：`fetch_log` 表中该 `(code, data_source)` 组合的 `MAX(end_date)`
- 仅 `status = 'success'` 的记录参与判断
- 中断后重新运行脚本，自动从断点继续

### 7.3 去重策略

- **股票列表去重**: 多源按 `code` 合并，保留信息最全的记录
- **日 K 数据去重**: 主键 `(code, trade_date, data_source)` + `INSERT OR IGNORE`
- **多源共存**: 同一股票同一天可存储多个源的数据，查询时可按 `data_source` 过滤

### 7.4 限流与容错

| 机制 | 实现 |
|------|------|
| 调用间隔 | `config.FETCH_DELAY_SECONDS`（默认 0.1s） |
| AkShare 重试 | 3 次重试，退避间隔 2s/4s/6s |
| 单股失败隔离 | 异常捕获 + 日志记录，不中断整体流程 |
| BaoStock 会话 | `login()/logout()` 管理，自动重连 |

## 11. AI Agent 短线分析系统

### 11.1 系统概述

在日 K 数据库基础上，构建了一个基于 LLM 的短线分析 AI Agent。Agent 通过 ReAct 循环自动调用工具查询数据库、计算技术指标、搜索新闻，并结合内置短线战法库和长期记忆系统，生成结构化的短线分析报告。

**核心特性**:
- **ReAct 循环**: 用户输入 → LLM 思考 → 调用工具 → 返回结果 → 继续/回答
- **双协议支持**: 同时兼容 OpenAI 和 Anthropic API 协议
- **流式输出**: 支持实时流式输出，包括 thinking 推理过程展示
- **多源新闻**: 纯爬虫方案获取百度/东方财富/新浪财经热点新闻
- **实时行情**: 新浪财经 + 东方财富双源冗余，自动故障切换与熔断保护
- **短线战法库**: 40 个完整战法（Markdown），根据技术面自动匹配
- **长期记忆**: 跨会话记住分析结论、用户偏好，支持向量语义搜索（sentence-transformers 768维）和 BM25 双模式
- **学习能力**: 自动检测用户教学/纠错，保存为结构化学习记忆
- **预测跟踪**: 保存投资预测并自动验证结果，从对错中持续学习
- **用户画像**: 自动提取交易风格、风险偏好、关注板块，个性化分析建议
- **知识图谱**: 结构化存储指标、策略、规则之间的关系，支持关联推理
- **LLM 记忆整合**: 会话结束时用 LLM 语义合并相似记忆，避免信息冗余
- **多周期分析**: 支持 1/5/15/30/60 分钟和周线数据，多周期共振分析
- **策略回测**: 基于历史数据验证策略的胜率、盈亏比、最大回撤、夏普比率
- **龙虎榜/北向资金**: 分析游资机构动向和外资态度
- **K线图生成**: 专业K线图和涨幅对比图，浏览器自动打开
- **自选股管理**: 分组管理自选股，设置目标价和止损价
- **盯盘告警**: 价格/涨跌幅/放量/RSI 多维度告警，桌面通知
- **交易日志**: 记录买卖操作，自动更新持仓，查看交易历史

#### 11.3.2 client.py - 统一 LLM 客户端

**核心类**: `LLMClient`

根据 `protocol` 字段自动选择 OpenAI 或 Anthropic SDK，提供统一的对话和流式输出接口。

| 方法 | 签名 | 说明 |
|------|------|------|
| `__init__` | `(setting=None)` | 加载配置并初始化 SDK 客户端 |
| `chat` | `(messages, tools=None) → dict` | 非流式对话，返回 `{content, tool_calls, stop_reason}` |
| `chat_stream` | `(messages, tools=None) → Generator[StreamChunk]` | 流式对话，yield `StreamChunk` 对象 |
| `format_assistant_message` | `(response) → dict` | 将响应转换为消息格式存入历史 |
| `format_tool_result_message` | `(tool_call_id, tool_name, result) → dict` | 构造工具结果消息 |

**StreamChunk 数据类**:

```python
@dataclass
class StreamChunk:
    type: str   # "thinking" / "content" / "tool_calls" / "done"
    text: str = ""
    tool_calls: list = field(default_factory=list)
```

**流式输出实现要点**:
- OpenAI 协议: 通过 `stream=True` 获取 SSE 流，累积 `delta.content`、`delta.tool_calls`、`delta.reasoning_content`（DeepSeek 思维链）
- Anthropic 协议: 通过 `client.messages.stream()` 获取事件流，处理 `content_block_start`、`content_block_delta`、`content_block_stop`
- 流完成后通过 `StopIteration.value` 捕获生成器 return 值，存入 `self.last_stream_response`

**DeepSeek thinking 模型特殊处理**:
- `reasoning_content` 字段需要在后续消息中回传，否则 API 报错
- `format_assistant_message()` 会将 `reasoning_content` 写入消息历史

#### 11.3.3 agent.py - Agent 核心

**核心类**: `StockAgent`

| 方法 | 说明 |
|------|------|
| `__init__(setting, verbose, session_id)` | 初始化 LLM 客户端、记忆系统、加载系统提示词 + 记忆上下文 + 战法列表 |
| `chat(user_input) → str` | 非流式对话，返回最终回复文本 |
| `chat_stream(user_input) → Generator` | 流式对话，yield `(type, text)` 元组 |
| `reset()` | 重置对话历史（先保存会话摘要，重新加载记忆上下文） |
| `on_session_end()` | 会话结束处理：运行衰减遗忘 + LLM 记忆整合 + 自动提取用户画像 + 保存摘要 |
| `_check_memory_update(tool_name, result)` | 记忆保存后注入 system 提示，learning 类型触发上下文刷新 |
| `refresh_memory_context()` | 重新查询 top 记忆并更新 system message |

**chat_stream 事件类型**:

| type | text 内容 | 说明 |
|------|-----------|------|
| `"round"` | 轮次数（int） | 标记当前是第几轮工具调用 |
| `"thinking"` | 推理文本片段 | LLM 的思考过程（DeepSeek 等支持） |
| `"content"` | 正文文本片段 | LLM 的回复内容 |
| `"tool_start"` | `工具名(参数JSON)` | 工具调用开始 |
| `"tool_result"` | 工具返回结果（截断至 300 字符） | 工具执行完成 |
| `"done"` | 空字符串 | 对话结束 |

**ReAct 循环流程**:

```
用户输入
  │
  ▼
┌──────────────────────────────────┐
│  Round N: 调用 LLM（流式输出）    │
│    ├─ thinking → 实时 yield       │
│    └─ content  → 实时 yield       │
│                                  │
│  流完成 → 获取结构化响应           │
│                                  │
│  if stop_reason == "stop":       │
│    yield done → 结束              │
│                                  │
│  if tool_calls:                  │
│    for each tool_call:           │
│      yield tool_start            │
│      执行工具 → yield tool_result │
│      结果写入消息历史              │
│    → 进入 Round N+1              │
└──────────────────────────────────┘
```

#### 11.3.5 prompts.py - 系统提示词

定义了 Agent 的角色、能力、分析框架、教学检测和输出格式：

- **角色**: 专业 A 股短线交易分析师
- **能力**: 查询 K 线、计算技术指标、搜索新闻、实时行情、数据更新、战法匹配、市场情绪分析、K线形态识别、股票筛选、预测跟踪、用户画像、知识图谱、多周期分析、策略回测、龙虎榜/北向资金、K线图生成、自选股管理、盯盘告警、交易日志、复盘系统
- **分析框架**: 技术面（量价、均线、MACD、RSI、KDJ、布林带）+ 形态识别 + 资金面 + 消息面（新闻事件）+ 战法规则 + 多周期共振 + 动态推理链（市场状态自适应）
- **数据更新策略**: 分析单只股票前先调用 `update_stock`，用户要求时调用 `update_all_stocks`
- **交易日概念**: `days` 参数指交易日（剔除周末和节假日），向用户解释时需说明
- **数据去重**: K线数据和实时行情可能包含同一天数据，优先使用实时行情
- **市场情绪分析**: 用 `market_sentiment` 判断情绪周期（高潮/发酵/启动/低迷/冰点），用 `limit_up_pool` 分析连板梯队，用 `sector_ranking` 发现热点板块
- **K线形态识别**: 用 `recognize_kline_patterns` 识别经典形态（锤子线、吞没、早晨之星等），结合量价和位置综合判断
- **股票筛选**: 用 `screen_stocks` 按技术条件从全市场筛选（MA、RSI、MACD、成交量、涨跌幅）
- **教学检测**: 自动识别用户纠错/教学信号，保存为 `learning` 类型记忆
- **记忆系统**: 分析前搜索历史记忆，完成后保存结论，检测教学时保存学习记忆
- **策略回测**: 用 `backtest_stock`/`backtest_multi_stock` 验证策略的胜率、盈亏比、最大回撤
- **龙虎榜/北向资金**: 用 `dragon_tiger_list`/`northbound_flow`/`sector_fund_flow` 分析资金动向
- **K线图生成**: 用 `generate_chart`/`compare_stocks_chart` 生成专业K线图和涨幅对比图
- **自选股/告警**: 用 `add_to_watchlist`/`set_alert_rule` 管理自选股和盯盘告警
- **交易日志**: 用 `record_trade`/`view_portfolio`/`generate_review` 记录交易和生成复盘报告
- **动态推理链**: 根据市场状态（高潮/发酵/启动/低迷/冰点）自动调整分析策略侧重点
- **预测跟踪**: 每次分析必须保存预测，分析同只股票时自动验证之前的预测
- **用户画像**: 分析前获取用户画像，根据交易风格调整建议
- **知识图谱**: 发现指标与战法关联时添加到知识图谱
- **输出要求**: 结构化表格、明确趋势判断、关键价位、操作建议、风险提示、免责声明

#### 11.3.7 memory.py - 长期记忆系统

v4.0 全面升级，包含 6 个核心类，实现向量语义搜索、预测跟踪、用户画像、知识图谱、LLM 整合。

**类结构**:

```
memory.py
├── EmbeddingIndex    — 向量 embedding 生成与检索
├── BM25Index         — BM25 关键词搜索（回退方案）
├── PredictionTracker — 投资预测跟踪与验证
├── UserProfile       — 用户交易画像管理
├── KnowledgeGraph    — 结构化知识图谱
└── MemoryManager     — 统一记忆管理（整合所有子系统）
```

##### EmbeddingIndex

负责向量 embedding 的生成、存储和检索。

| 方法 | 说明 |
|------|------|
| `__init__(setting?)` | 初始化 embedding 提供商（local/API/none 三级回退） |
| `encode_single(text) → list[float]` | 单条文本编码为 768 维向量 |
| `encode_batch(texts) → list[list[float]]` | 批量编码（每批最多 10 条，间隔 0.5s） |
| `search(query_vec, candidates, top_k) → list` | 余弦相似度搜索 |
| `vector_to_bytes(vec) → bytes` | 向量序列化为 BLOB（DuckDB 存储） |
| `bytes_to_vector(blob) → list[float]` | BLOB 反序列化为向量 |

**三级回退策略**:
1. `provider: "openai"` — 调用 API 生成 embedding（需配置 api_key/base_url）
2. `provider: "local"` — 本地 sentence-transformers 模型（零费用，首次自动下载 ~400MB）
3. `provider: "none"` — 无 embedding，降级为 BM25

**本地模型**: `shibing624/text2vec-base-chinese`（768 维），自动设置 `HF_ENDPOINT=https://hf-mirror.com` 加速国内下载。

##### BM25Index

无 embedding 时的关键词搜索方案，使用 `rank_bm25` 库实现 BM25 算法。

| 方法 | 说明 |
|------|------|
| `build(documents)` | 构建 BM25 索引（文档列表：id, text, keywords） |
| `search(query, query_keywords, top_k) → list` | BM25 搜索，返回 (id, score) 列表 |

**中文分词**: 使用正则 `[一-鿿]+|[a-zA-Z]+|\d+` 切分，不依赖 jieba 等外部库。

##### PredictionTracker

投资预测的保存、验证和准确率统计。

| 方法 | 说明 |
|------|------|
| `save_prediction(stock_code, stock_name, prediction_type, direction, target_price, stop_loss, timeframe_days, reasoning, confidence) → str` | 保存预测（JSON） |
| `check_predictions(stock_code?) → str` | 查询待验证预测 |
| `verify_prediction(prediction_id, actual_price, days_elapsed?) → str` | 验证预测，自动调整关联记忆 importance |
| `get_accuracy_stats() → str` | 预测准确率统计（总数、正确率、按方向分类） |

**自动学习机制**:
- 正确预测 → 关联记忆 importance +0.1
- 错误预测 → 关联记忆 importance -0.1

##### UserProfile

用户交易画像的自动提取和管理。

| 方法 | 说明 |
|------|------|
| `get_profile() → str` | 获取完整用户画像（JSON） |
| `update_profile(key, value, confidence?) → str` | 更新画像 |
| `get_summary_text() → str` | 获取画像摘要文本（注入 system prompt） |
| `extract_from_conversation(messages) → list` | 从对话中自动提取画像信号 |

**画像维度**: trading_style(交易风格)、risk_tolerance(风险偏好)、stop_loss_habit(止损习惯)、preferred_indicators(偏好指标)、watched_sectors(关注板块)、watched_stocks(关注股票)

**自动提取规则**: 基于正则匹配，如"低吸" → trading_style: 低吸型，"止损设5%" → stop_loss_habit: 5%

##### KnowledgeGraph

结构化交易知识图谱，管理实体和关系。

| 方法 | 说明 |
|------|------|
| `add_entity(entity_type, name, attributes?) → str` | 添加实体（返回 ID） |
| `add_relation(source_id, target_id, relation_type, weight?) → str` | 添加关系 |
| `query_related(query?, entity_type?) → str` | 查询相关实体关系 |
| `extract_from_memory(content) → tuple` | 从记忆文本自动提取实体和关系 |

**实体类型**: stock(股票)、indicator(指标)、pattern(形态)、strategy(策略)、concept(概念)、rule(规则)

**关系类型**: triggers(触发)、requires(需要)、contradicts(矛盾)、supports(支持)、applies_to(适用于)、user_prefers(用户偏好)

##### MemoryManager

统一记忆管理器，整合所有子系统。

| 方法 | 说明 |
|------|------|
| `save_memory(content, memory_type, keywords, tags, importance, ...)` | 保存记忆（自动生成 embedding + 提取知识实体） |
| `search_memories(query, memory_type, top_k)` | 搜索记忆（自动选择向量/BM25/关键词） |
| `_search_with_embedding(query, ...)` | 向量搜索：粗排 top-50 → 关键词精排 |
| `_search_with_bm25(query, ...)` | BM25 搜索 |
| `get_context_block(query_text, max_chars)` | 智能上下文注入（多策略预算分配） |
| `compress_memories_with_llm(client)` | LLM 驱动的记忆语义合并 |
| `compress_memories(target_type, max_merge)` | 规则压缩（兼容旧版） |
| `decay_and_forget(threshold)` | 归档低权重记忆 |
| `detect_learning_signals(messages)` | 检测用户教学/纠错信号（双层：规则+prompt） |
| `update_memory(memory_id, ...)` | 更新记忆字段 |
| `delete_memory(memory_id)` | 软删除（归档） |
| `list_memories(memory_type, limit)` | 列出记忆 |
| `start_session(session_id)` | 记录会话开始 |
| `end_session(session_id, summary, ...)` | 结束会话，保存摘要 |

**智能上下文注入（预算分配）**:
```
上下文预算: 4000 字符（可配置）
  1. [30%] 当前查询最相关的记忆（向量检索 top-3）
  2. [25%] 用户教学/纠错记忆（learning 类型优先）
  3. [20%] 用户画像摘要（交易风格、偏好）
  4. [15%] 最近会话摘要（跨会话连续性）
  5. [10%] 高置信度的预测结果（正确/错误的教训）
多样性控制: 同一股票的记忆最多占 40%
```

**LLM 记忆整合**:
- 触发条件: 会话结束时，本轮新增 >= 3 条同类型记忆
- 流程: 收集同类记忆 → LLM 语义合并 → 替换原始记忆 → 标记被合并项为 archived
- 每次最多合并 5 条，max_tokens=2000

**学习信号检测（双层机制）**:
- 规则层: 正则匹配否定词+纠正内容、"记住"/"以后"/"注意"等教学信号
- LLM 层: prompt 指令引导 Agent 主动调用 `record_learning`

**记忆类型与半衰期**:

| 类型 | 用途 | 半衰期 |
|------|------|--------|
| `preference` | 用户偏好 | 90 天 |
| `insight` | 分析结论 | 30 天 |
| `fact` | 重要事实 | 60 天 |
| `context` | 上下文线索 | 14 天 |
| `summary` | 会话摘要 | 45 天 |
| `learning` | 用户教学/纠错 | 180 天 |

**评分公式**:
```
# 向量搜索模式
score = 0.6 * cosine_similarity + 0.25 * keyword_overlap + 0.15 * effective_weight

# BM25/关键词模式
score = effective_weight * (0.5 * keyword_overlap + 0.3 * tag_overlap + 0.2 * stock_code_match)

effective_weight = importance * exp(-0.693 * days_since_last_access / half_life)
```

**learning 类型结构化字段**:
- `learned_what`: 学到了什么（简明扼要）
- `learned_why`: 为什么重要
- `apply_when`: 什么情况下应用

**自动维护**:
- 会话结束时自动运行 `decay_and_forget()` + `compress_memories_with_llm()`
- 记忆保存后注入 system 提示，learning 类型触发完整上下文刷新
- 保存记忆时自动提取知识图谱实体和关系

**告警类型**: `price`(价格)、`change_pct`(涨跌幅)、`volume`(放量)、`rsi`(RSI值)。

#### 11.3.15 journal.py - 交易日志与复盘系统

**核心函数**:

| 函数 | 说明 |
|------|------|

#### 11.3.16 reasoning.py - 动态推理链

根据市场状态自动调整分析策略的推理引擎。

**市场状态分类**:
- **高潮期**: 涨停数多、连板高度大 → 重点分析高位接力风险
- **发酵期**: 热点初现、板块轮动 → 重点发掘低位首板机会
- **启动期**: 情绪回暖、赚钱效应初现 → 重点关注板块龙头
- **低迷期**: 涨停稀少、赚钱效应差 → 提示风险，建议轻仓
- **冰点期**: 极度低迷 → 建议空仓等待

根据不同的市场状态，自动调整分析框架的侧重点和操作建议的激进程度。

```python
from datetime import date
import pandas as pd
from .base import DataSourceBase


class TushareSource(DataSourceBase):

    def __init__(self, token: str):
        import tushare as ts
        self.pro = ts.pro_api(token)

    @property
    def name(self) -> str:
        return "tushare"

    def fetch_stock_list(self) -> list[dict]:
        """返回格式必须包含: code, name, market, board, list_date"""
        df = self.pro.stock_basic(exchange='', list_status='L',
                                  fields='ts_code,name,market,list_date')
        results = []
        for _, row in df.iterrows():
            code = row["ts_code"][:6]
            if code.startswith("6") and not code.startswith("688"):
                market = "sh"
            elif code.startswith("0"):
                market = "sz"
            else:
                continue
            results.append({
                "code": code,
                "name": row["name"],
                "market": market,
                "board": "main",
                "list_date": row["list_date"],
            })
        return results

    def fetch_daily_kline(self, code: str, start_date: date,
                          end_date: date) -> pd.DataFrame:
        """返回格式必须包含: trade_date, open, high, low, close, volume, amount, turnover"""
        ts_code = f"{code}.SH" if code.startswith("6") else f"{code}.SZ"
        df = self.pro.daily(ts_code=ts_code,
                            start_date=start_date.strftime("%Y%m%d"),
                            end_date=end_date.strftime("%Y%m%d"))
        if df.empty:
            return pd.DataFrame()

        df = df.rename(columns={
            "trade_date": "trade_date",
            "open": "open", "high": "high",
            "low": "low", "close": "close",
            "vol": "volume", "amount": "amount",
        })
        df["trade_date"] = pd.to_datetime(df["trade_date"]).dt.date
        df["code"] = code
        df["turnover"] = None  # Tushare daily 无换手率
        df["data_source"] = self.name
        return df[["code", "trade_date", "open", "high", "low", "close",
                    "volume", "amount", "turnover", "data_source"]]
```

### 12.3 添加技术指标计算

在 `query.py` 中扩展 `StockQuery` 类：

```python
def get_ma(self, code: str, windows: list[int] = [5, 10, 20, 60],
           start_date: str = None, end_date: str = None) -> pd.DataFrame:
    """计算移动平均线"""
    df = self.get_daily_kline(code, start_date, end_date)
    for w in windows:
        df[f"ma{w}"] = df["close"].rolling(window=w).mean()
    return df

def get_macd(self, code: str, start_date: str = None,
             end_date: str = None) -> pd.DataFrame:
    """计算 MACD"""
    df = self.get_daily_kline(code, start_date, end_date)
    ema12 = df["close"].ewm(span=12).mean()
    ema26 = df["close"].ewm(span=26).mean()
    df["dif"] = ema12 - ema26
    df["dea"] = df["dif"].ewm(span=9).mean()
    df["macd"] = (df["dif"] - df["dea"]) * 2
    return df

def get_rsi(self, code: str, period: int = 14,
            start_date: str = None, end_date: str = None) -> pd.DataFrame:
    """计算 RSI"""
    df = self.get_daily_kline(code, start_date, end_date)
    delta = df["close"].diff()
    gain = delta.where(delta > 0, 0).rolling(window=period).mean()
    loss = (-delta.where(delta < 0, 0)).rolling(window=period).mean()
    df["rsi"] = 100 - (100 / (1 + gain / loss))
    return df
```

### 12.5 直接使用 DuckDB SQL

DuckDB 支持丰富的 SQL 语法，适合复杂分析场景：

```python
import duckdb
conn = duckdb.connect("db/astock.duckdb")

# 连续上涨天数
conn.execute("""
    WITH daily AS (
        SELECT code, trade_date, close,
               LAG(close) OVER (PARTITION BY code ORDER BY trade_date) AS prev_close
        FROM daily_kline WHERE data_source = 'baostock'
    ),
    streaks AS (
        SELECT code, trade_date, close,
               SUM(CASE WHEN close > prev_close THEN 0 ELSE 1 END)
               OVER (PARTITION BY code ORDER BY trade_date) AS grp
        FROM daily
    )
    SELECT code, COUNT(*) AS up_days
    FROM streaks
    WHERE close > prev_close
    GROUP BY code, grp
    ORDER BY up_days DESC
    LIMIT 10
""").fetchdf()

# 量价齐升股票
conn.execute("""
    SELECT k.code, s.name, k.trade_date, k.close, k.volume
    FROM daily_kline k
    JOIN stock_info s ON k.code = s.code
    WHERE k.trade_date = current_date - INTERVAL '1' DAY
      AND k.close > LAG(k.close) OVER (PARTITION BY k.code ORDER BY k.trade_date)
      AND k.volume > LAG(k.volume) OVER (PARTITION BY k.code ORDER BY k.trade_date)
    ORDER BY k.volume DESC
    LIMIT 20
""").fetchdf()

# 与 pandas DataFrame 互操作
df = conn.execute("SELECT * FROM daily_kline WHERE code = '600000'").fetchdf()
```

## 14. 已知问题与限制

### 14.1 当前限制

| 问题 | 说明 | 影响 |
|------|------|------|
| 单线程采集 | 当前串行遍历股票，未使用并发 | 全量采集耗时约 2 小时 |
| AkShare 不稳定 | 部分网络环境下连接频繁断开 | 建议主要使用 BaoStock |
| 无自动定时 | 需手动运行采集脚本 | 需要自行配置定时任务 |
| 仅日 K 级别 | 不支持分钟级/Level2 数据 | 不适合高频策略 |
| 前复权价格 | 只存前复权，不存原始价格 | 除权后历史价格会变化 |
| 新闻爬虫稳定性 | 依赖网页结构，网站改版可能导致失效 | 需定期维护选择器 |

### 14.2 技术债务

1. `database.py` 中 `fetch_log` 的 ID 使用 `MAX(id)+1` 而非自增序列，高并发下可能冲突（当前单线程无影响）

3. `StockQuery` 每次调用都新建连接，高频调用时应考虑连接池
